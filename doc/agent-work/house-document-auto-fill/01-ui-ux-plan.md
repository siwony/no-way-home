# UI/UX Plan: 등기부등본·임대차 계약서 자동 입력

Status: READY_FOR_DIRECTOR_REVIEW

## User Goal

사용자는 기존 주택 계약 위험 진단 워크스페이스 안에서 등기부등본과 임대차 계약서를 등록하고, 시스템이 추출한 핵심 값을 문서 근거와 함께 검토한 뒤, 승인한 필드만 기존 계약 정보 및 등기 확인 입력값에 반영해 분석까지 이어갈 수 있어야 한다.

## Primary Flow

1. 사용자는 현재 운영형 단일 워크스페이스에서 `User ID`를 적용하고 진단 작업을 시작한다.
2. 사용자는 `문서 자동 입력` 카드에서 등기부등본 PDF와 임대차 계약서 PDF 또는 이미지를 각각 전용 슬롯에 업로드한다.
3. 시스템은 문서별 상태를 `UPLOADED -> EXTRACTING -> REVIEW_REQUIRED`로 갱신하고, 각 문서 카드에서 진행률, 마지막 갱신 시각, 실패 여부를 보여준다.
4. 한 문서라도 `REVIEW_REQUIRED`가 되면 사용자는 `추출 검토` 카드에서 필드별 후보값, 신뢰도, 문서 근거, 문서 간 불일치 경고를 확인한다.
5. 사용자는 각 필드를 `수정`, `제외`, `승인`할 수 있다. 승인 전에는 기존 계약 기본 정보와 등기 수기 확인 값이 바뀌지 않는다.
6. 사용자가 `승인한 필드 반영`을 실행하면 승인된 필드만 기존 `계약 기본 정보`와 `등기 확인` 입력 영역에 채워지고, 각 입력은 `자동 입력 반영됨` 출처 표시와 함께 계속 수동 편집 가능해야 한다.
7. 사용자는 자동 입력으로 채워지지 않았거나 제외한 값만 보완하고, 기존 `건축물대장 확인`, `시세 입력`, `분석 실행` 흐름으로 이어간다.
8. 문서 추출이 실패하거나 검토를 중단해도 사용자는 `수기 입력으로 계속`을 통해 기존 수동 입력 흐름으로 복귀할 수 있어야 한다.

## UX Principles

- 현재 `frontend/src/App.tsx`의 단일 operational workspace, 카드, status rail, 배너, 상태 pill 패턴을 유지한다.
- 자동 추출은 `초안`으로만 취급한다. 승인 전 자동 반영, 숨은 overwrite, 자동 분석 확정은 금지한다.
- 문서별 상태와 필드별 상태를 분리한다. 문서가 `REVIEW_REQUIRED`여도 일부 필드는 이미 승인되었을 수 있어야 한다.
- 문서 간 불일치와 낮은 신뢰도는 눈에 띄게 보여주되, 법률 판단이나 안전 확정 문구로 이어지지 않게 한다.
- 민감 문서이므로 다운로드 유도보다 `필요 최소한의 미리보기 + 근거 확인` 중심으로 설계한다.
- 추출 실패 시 막다른 길을 만들지 않는다. 항상 수기 입력 경로와 재시도 경로를 남긴다.

## Recommended Workspace Structure

- `세션 / 접근 경계 바`
  - 기존 `User ID`, `checkId`, `새 진단 시작` 패턴 유지
  - 접근 거부 또는 경계 전환 메시지는 현재 global banner 패턴을 재사용
- `진행 상태 rail`
  - 권장 항목: `계약 기본 정보`, `등기 문서`, `계약서 문서`, `추출 검토`, `승인 반영`, `건축물 확인`, `시세 입력`, `분석`
  - 문서 상태는 `미업로드`, `업로드됨`, `추출 중`, `검토 필요`, `승인 완료`, `실패`
- `문서 자동 입력 카드`
  - 하위 2개 문서 슬롯: `등기부등본`, `임대차 계약서`
  - 기존 문서 업로드 카드보다 상위 개념으로 배치하고, 기존 수기 입력 카드보다 먼저 배치
- `추출 검토 카드`
  - 문서 탭, 불일치 경고, 필드 테이블, 근거 drawer 또는 inline evidence panel 포함
- `기존 입력 카드`
  - `계약 기본 정보`와 `등기 확인`은 자동 입력 반영 대상
  - `건축물대장 확인`과 `시세 입력`은 첫 slice에서 기존 수동 흐름 유지
- `분석 및 결과 카드`
  - 기존 구조 유지
  - 자동 입력 출처와 확인 필요 항목이 분석 전에도 보이도록 연결

## Detailed Interfaces

- `문서 자동 입력 카드`
  - 제목: `문서 자동 입력`
  - helper copy: `등기부등본과 임대차 계약서를 등록하면 핵심 항목을 추출해 검토용 초안을 만듭니다. 승인 전에는 진단 입력값에 반영되지 않습니다.`
  - privacy copy: `문서 원본과 추출 결과는 민감 정보로 처리됩니다. 인터넷등기소 로그인 정보, 공동인증서 비밀번호, 결제 정보는 입력하거나 저장하지 않습니다.`
  - 문서 슬롯 1: `등기부등본 PDF`
  - 문서 슬롯 2: `임대차 계약서 PDF / 이미지`
  - 슬롯별 표시 요소:
    - 상태 pill
    - 파일명
    - 업로드 시각 또는 마지막 처리 시각
    - 지원 형식 안내
    - `업로드`, `다시 업로드`, `삭제`, `다시 처리` 액션
  - 문서 종류는 고정 슬롯 기준으로 받는다. 자동 분류는 보조 검증으로만 쓰고, 예상 문서와 다르면 `문서 종류가 예상과 다를 수 있습니다.` 경고를 띄운다.

- `추출 검토 카드`
  - 제목: `추출 검토 및 승인`
  - 상단 요약:
    - `검토 필요 n건`
    - `낮은 신뢰도 n건`
    - `문서 불일치 n건`
    - `승인 완료 n건`
  - 문서 탭:
    - `전체`
    - `등기부등본`
    - `임대차 계약서`
    - `불일치`
  - 필드 row 권장 컬럼:
    - 필드명
    - 현재 후보값 또는 편집 입력
    - 신뢰도 chip
    - 출처 문서와 페이지
    - 원문 근거 보기
    - 상태
    - 액션
  - 필드 액션:
    - `수정`
    - `제외`
    - `승인`
    - 수정 중이면 `수정 취소`
  - 근거 보기:
    - 최소 정보: `문서명`, `페이지`, `원문 발췌`
    - 가능하면 현재 카드 안의 expandable panel 또는 side drawer 사용
    - 새 route나 전체 화면 viewer는 첫 slice 필수 아님
  - 핵심 규칙:
    - `주소`, `임대인`, `소유자`, `보증금`, `월세`, `계약일`, `권리관계 위험 항목`은 신뢰도가 높아도 기본 상태를 `검토 필요`로 둔다.
    - `특약 핵심 문장 후보`는 review-only 영역에 표시하고 첫 slice에서는 구조화 입력값으로 자동 반영하지 않는다.
    - `임차인명`, 주민등록번호, 계좌번호 등 민감 정보는 저장 제외 또는 마스킹된 근거만 노출한다.

- `불일치 경고 영역`
  - 위치: `추출 검토 카드` 최상단
  - 우선 노출 항목:
    - 계약서 주소 vs 등기부등본 주소
    - 계약서 임대인 vs 등기부등본 소유자
    - 계약서 보증금 vs 현재 계약 기본 정보
    - 등기부등본 발급일 또는 열람일 신선도 경고
  - 각 경고는 `차이 설명`, `왜 확인이 필요한지`, `관련 필드로 이동` 액션을 가진다.
  - 문구는 `자동으로 잘못되었다고 확정하지 않고 확인이 필요함`을 유지한다.

- `승인 반영 요약 영역`
  - 위치: `추출 검토 카드` 하단 또는 기존 입력 카드 상단
  - 항목:
    - `반영 예정 필드`
    - `수동 값과 충돌하는 필드`
    - `제외된 필드`
  - 메인 CTA:
    - `승인한 필드 반영`
    - `지금은 수기 입력으로 계속`
  - overwrite 규칙:
    - 기존 입력값이 비어 있으면 바로 반영
    - 기존 입력값이 이미 있으면 `현재 값 유지`와 `승인값으로 교체`를 비교 요약해 보여준 뒤 반영

- `기존 입력 카드 반영 방식`
  - 기존 `계약 기본 정보`와 `등기 수기 확인` 카드 자체는 유지한다.
  - 반영된 필드는 읽기 전용으로 잠그지 않는다. 사용자가 이후 다시 수정할 수 있어야 한다.
  - 각 반영 필드에는 source chip을 붙인다.
  - 권장 chip:
    - `자동 입력 반영`
    - `사용자 수정됨`
    - `검토 필요`
  - 사용자가 반영 후 다시 값을 바꾸면 원래 추출값과의 연결은 남기되, 상태는 `사용자 수정됨`으로 바뀌어야 한다.

## Field Application Map

- 임대차 계약서에서 기존 `계약 기본 정보`로 반영되는 후보:
  - `addressRoad`
  - `addressLot`
  - `contractType`
  - `depositAmount`
  - `monthlyRentAmount`
  - `contractPlannedDate`
  - `occupancyPlannedDate`
  - `landlordName`
- 등기부등본에서 기존 `등기 확인`으로 반영되는 후보:
  - `currentOwnerName`
  - `ownerMatchesLandlord`
  - `hasTrustRegistration`
  - `hasSeizure`
  - `hasProvisionalSeizure`
  - `hasProvisionalDisposition`
  - `hasAuctionProceeding`
  - `hasLeaseRegistration`
  - `hasMortgage`
  - `seniorDebtAmount`
- 문서에서 보여주되 첫 slice에서 자동 반영하지 않는 항목:
  - 등기부등본 발급일 또는 열람일
  - 계약서 특약 핵심 문장 후보
  - 임차인명 마스킹 값
  - 문서 분류 경고, 주소 정규화 경고, 최신성 경고

## States

### Normal And Partial States

- `두 문서 모두 미업로드`
  - 안내: `등기부등본과 임대차 계약서를 등록하면 자동 입력 검토를 시작할 수 있습니다.`
- `한 문서만 업로드`
  - 안내: `현재 문서만 먼저 처리할 수 있습니다. 다른 문서는 나중에 추가해도 됩니다.`
- `한 문서는 승인 완료, 다른 문서는 검토 필요`
  - 허용: 부분 승인 반영
  - 표시: `일부 필드만 반영됨. 남은 문서는 계속 검토할 수 있습니다.`

### Loading And Processing

- 업로드 중
  - 버튼 라벨: `업로드 중...`
  - 같은 문서 슬롯의 다른 액션만 비활성화
- 추출 중
  - 문서 카드 상태: `추출 중`
  - 안내: `문서에서 핵심 항목을 찾고 있습니다. 완료되면 검토가 필요합니다.`
  - progress가 있으면 `%`보다 단계명 중심으로 보여준다. 예: `텍스트 추출 완료 · 필드 정리 중`
- 승인 반영 중
  - CTA 라벨: `반영 중...`
  - 안내: `승인한 필드를 기존 입력값에 반영하고 있습니다.`

### Review Required

- 문서 상태 `REVIEW_REQUIRED`
  - 문서 카드에 `검토 필요` pill
  - 추출 검토 카드가 기본 펼침 상태로 열린다.
- 필드 상태 `검토 필요`
  - 낮은 신뢰도, 핵심 필드, 문서 불일치 관련 필드에 기본 적용
- 필드 상태 `수정 중`
  - 값 입력란, 원래 추출값, 근거 발췌를 함께 보여준다.

### Empty

- 추출 결과 없음
  - `아직 검토할 추출 결과가 없습니다. 문서를 업로드하거나 처리 완료를 기다려 주세요.`
- 승인할 필드 없음
  - `이번 문서에서는 바로 반영할 필드를 찾지 못했습니다. 필요한 값은 수기로 입력할 수 있습니다.`

### Success

- 업로드 성공
  - `문서가 등록되었습니다. 추출이 시작되면 상태가 자동으로 바뀝니다.`
- 추출 완료
  - `자동 입력 초안이 준비되었습니다. 반영 전에 내용을 검토해 주세요.`
- 승인 반영 완료
  - `승인한 필드만 기존 입력값에 반영했습니다. 필요하면 계속 수정할 수 있습니다.`
- 수기 복귀
  - `문서 자동 입력 없이 계속 진행합니다. 필요한 값은 직접 입력해 주세요.`

### Validation

- 지원하지 않는 파일 형식
  - `지원하지 않는 형식입니다. 등기부등본은 PDF, 계약서는 PDF 또는 지원 이미지 형식만 등록할 수 있습니다.`
- 파일 없음
  - `업로드할 파일을 먼저 선택해 주세요.`
- 문서 종류 불일치
  - `업로드한 파일이 선택한 문서 종류와 다를 수 있습니다. 문서를 다시 확인해 주세요.`
- 승인 없이 반영 시도
  - `승인한 필드가 없습니다. 반영할 항목을 먼저 확인해 주세요.`

### Error And Recovery

- 암호화된 PDF
  - `암호화된 PDF는 바로 읽을 수 없습니다. 비밀번호 없이 열리는 파일로 다시 업로드해 주세요.`
- 이미지 품질 부족
  - `문자 인식이 어려웠습니다. 더 선명한 파일로 다시 업로드하거나 수기로 입력해 주세요.`
- 추출 실패
  - `자동 추출을 완료하지 못했습니다. 다시 처리하거나 수기 입력으로 계속할 수 있습니다.`
- provider 또는 처리 장애
  - `문서 처리에 일시적인 문제가 있습니다. 잠시 후 다시 시도해 주세요.`
- 삭제됨
  - `문서가 삭제되어 원문과 추출 결과를 더 이상 사용할 수 없습니다. 필요하면 다시 업로드해 주세요.`

### Permission Denied

- 다른 사용자 접근 또는 권한 없음
  - `현재 User ID로는 이 문서와 추출 결과에 접근할 수 없습니다.`
  - 허용 액션: `User ID 다시 적용`, `새 진단 시작`
  - 금지: 기존 문서 파일명, 추출값, 원문 근거를 화면에 남겨두는 것

## Developer Notes

- 기존 `App.tsx` 구조를 기준으로 새 기능을 별도 페이지로 분리하지 말고, 현재 카드 기반 워크스페이스 안에 단계형 섹션으로 추가하는 편이 적합하다.
- 구현에서는 최소 세 층을 분리해야 한다: `문서 처리 상태`, `추출된 원본 후보값`, `사용자 검토 후 승인된 값`.
- 승인된 값이 실제 기존 form state에 반영되기 전까지는 현재 `contractForm`과 `registryFindingsForm`을 직접 mutate하지 않는 구조가 안전하다.
- 문서 미리보기는 `다운로드`보다 `근거 발췌 + 페이지 정보 + 제한된 미리보기` 중심으로 시작하는 편이 보안과 구현 복잡도 모두에서 유리하다.
- 기존 상태 pill, banner, sub-card 패턴을 재사용하면 새 컴포넌트가 들어와도 시각 일관성을 유지하기 쉽다.
- `건축물대장 확인`과 `시세 입력`은 이번 scope에서 수동 단계로 남겨 두고, auto-fill review 흐름에 억지로 포함하지 않는다.
- 사용자가 이미 입력한 값을 auto-fill 승인값으로 덮어쓸 때는 조용히 overwrite하지 말고 반드시 비교 요약을 거친다.
- 추출 polling 또는 상태 갱신이 필요하면 카드 단위 loading과 refresh로 제한하고, 전체 화면 blocking 처리나 과한 실시간 애니메이션은 피한다.

## Open Questions

- 진단 작업 생성 시점을 어떻게 정의할지 결정이 필요하다. 자동 입력이 계약 기본 정보를 채우려면 `계약 기본 정보 입력 전 draft check 생성`이 필요할 수 있는데, 현재 brief는 이 진입 방식을 명시하지 않는다.
- 임대차 계약서 이미지 업로드의 정확한 지원 형식을 확정해야 한다. 첫 slice에서 `jpg`, `jpeg`, `png`, `webp`, `heic` 중 어디까지 허용할지 결정이 필요하다.
- 문서 보관 및 삭제 정책의 기본 사용자 문구가 필요하다. 예를 들어 `업로드 후 며칠 보관`, `승인 후 즉시 삭제 가능` 같은 정책이 정해져야 안내 copy를 확정할 수 있다.
- 업로드 후 원본 문서를 다시 열어 보는 범위를 어디까지 허용할지 결정이 필요하다. 리뷰 시점의 제한 미리보기만 허용할지, 이후 재열람 또는 다운로드를 허용할지에 따라 UI가 달라진다.
- 계약서에서 추출되는 임차인명과 특약 문장을 얼마나 저장할지 제품 판단이 필요하다. 현재 권장안은 `임차인명은 마스킹 또는 저장 제외`, `특약은 review-only 후보`이다.
