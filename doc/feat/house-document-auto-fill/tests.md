# Test Requirements

## Backend Tests

- 문서 업로드 후 `UPLOADED` 상태와 암호화 저장이 생성된다.
- 지원하지 않는 문서 형식은 명확한 validation error를 반환한다.
- OCR/extraction fake adapter 성공 시 추출 필드와 원문 근거가 저장된다.
- extraction 실패 시 `FAILED` 상태와 민감정보 없는 실패 사유가 저장된다.
- 사용자 승인 전에는 기존 진단 입력값이 확정 변경되지 않는다.
- 승인 후 승인된 필드만 기존 house check 입력 모델에 반영된다.
- 다른 `X-User-Id`는 문서와 추출 결과를 읽거나 승인할 수 없다.
- 원본 삭제 요청 후 문서 원본과 추출 artifact가 접근 불가 상태가 된다.

## Frontend Tests

- 문서 업로드, 처리 중, 검토 필요, 승인, 실패 상태가 구분되어 보인다.
- 추출 필드는 원문 근거, 신뢰도, 수정 입력, 제외 액션을 제공한다.
- 낮은 신뢰도와 불일치 항목이 명확하게 표시된다.
- 승인 전에는 기존 분석 입력에 반영되지 않는다.
- 승인 후 자동 입력 출처가 표시된다.
- 추출 실패 시 기존 수기 입력으로 계속 진행할 수 있다.

## E2E Tests

- 등기부등본 fixture와 임대차 계약서 fixture 업로드 후 추출 검토 화면이 열린다.
- 사용자가 일부 필드를 수정하고 승인하면 계약 기본 정보와 등기 확인 값이 채워진다.
- 계약서 임대인과 등기부등본 소유자 불일치 fixture에서 확인 필요 상태가 표시된다.
- 다른 사용자로 같은 checkId/documentId 접근 시 `ACCESS_DENIED`가 표시된다.
- 문서 원본/추출 결과가 브라우저 영구 저장소에 남지 않는다.

## Local-Only User Documents

사용자가 실제 검증용으로 다음 파일을 로컬 workspace 루트에 제공했다.

- `6_부동산_등기사항_전부증명서.pdf`
- `임대차계약서.pdf`

이 파일들은 민감 문서일 수 있으므로 커밋하지 않는다. QA는 mock fixture 기반 자동화와 별도로, 로컬에서 이 파일들을 업로드해 추출 결과와 검토 UI가 동작하는지 확인한다.
